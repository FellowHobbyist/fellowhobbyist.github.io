Summary

This post is a simple guide to quickly get up-and-running with Hugo and GitHub Pages for hosting your next static website over HTTPS via a Custom Domain.

We’ll also be working with Git Submodules. As it often happens while working on one project, you will need to use another project from within it. We will be using Git Submodules to accomplish the following:

  • Manage the Source Code via a Private Git Repository
  • Publish the Static Site to a separate GitHub Pages Repository
  • Source a Custom Hugo Theme

You can learn More About Git Submodules Here

What is Hugo?

Hugo is a static site generator written Go. Simply put, Hugo takes a source directory of files and templates and uses these as inputs to create a complete website.

Install Hugo

Hugo is written in Go, but you donâ€™t need to install Go to enjoy Hugo. We’ll just need to download a precompiled binary to get started.

Hugo is Cross-Platform, so you simply need to download the appropriate version for your platform. Once downloaded, the Hugo binary can be run from anywhere. Hugo provides pre-built binaries for all the major platforms including:

  • Linux
  • Mac
  • Windows

See Installation Docs or View all the Latest Releases Here

# VERIFY INSTALL
$ which hugo
/usr/local/bin/hugo

# VERIFY VERSION
$ hugo version
Hugo Static Site Generator v0.51 linux/amd64 BuildDate: 2018-11-07T10:10:13Z

Ok, now that Hugo is installed let’s get started with building our new site.

Create a New Project

Ok, let’s create a new hugo project:

# CREATE A NEW SITE
hugo new site blog

Pretty simple. Now let’s get things configured.

Configure your Project

Install a Theme

First, let’s install a theme. There are some awesome themes available. One of my favorite Hugo Themes is Mainroad and we are going to add this theme to our project as a git submodule:

# CHANGE INTO YOUR PROJECT'S ROOT DIRECTORY
cd blog   

# ADD THE THEME AS A GIT SUBMODULE (/themes/mainroad)
git submodule add https://github.com/Vimux/mainroad.git themes/mainroad;

Ok, you should now see themes/mainroad directory populated in your project. Be sure to take a look at .gitmodules and .git/modules in your root project to understand how this is configured.

Update config.toml

Next, let’s configure our config.toml file to set things up using using Mainroad’s config.toml example

Create a Blog Post

Next, let’s create a new post.

# CREATE A NEW BLOG POST
hugo new post/my-first-post.md

This will create a new markdown file at /blog/content/post/my-first-post.md

Now, let’s update the new post with Mainroad’s front matter example. Using “Front Matter” we can customize how our post is categorized, tagged, and displayed via the theme. Learn more about themes here.

Runing the Hugo Server

Now we are ready to view our work and iterate on our project.

# Run Hugo Server and Include Draft Content using the -D flag
hugo server -D

You should be able to navigate to localhost:1313 and view your new site. Hugo comes with LiveReload built in. There are no additional packages to install. As you make changes and save them, the Hugo Server will watch for changes and will use LiveReload to update the browser.

Using git submodules

Next, let’s create new GitHub Repository (e.g. https://github.com/username/username.github.io) to host our GitHub Pages Site.

Once the new github repository is created, let’s add it as a submodule

# ADD THE GITHUB PAGES REPOSITORY AS SUBMODULE (/public)
git submodule add https://github.com/FellowHobbyist/fellowhobbyist.github.io.git public

With this submodule configuration in place, we can now use Hugo to build our static site in the /public directory of our private git repository and then publish /public folder to the master branch of our public GitHub Pages respository.

Instead of doing this manually each time we want to build and deploy our site, how about we write a quick deploy script?

Create a Deployment Script

We need a consistent way to build and deploy our project to our GitHub Pages Repository. To accomplish this we are going to use a bash script (deploy.sh) that is saved in the root directory of our project. Be sure to update the permissions so that it is executable.

$ chmod 0755 ./blog/deploy.sh

Here is the script /blog/deploy.sh

#!/bin/bash

echo -e "\033[0;32mDeploying Blog to GitHub...\033[0m"

# Build the project.
echo "Building the Project... "
hugo # if using a theme, replace with `hugo -t <YOURTHEME>`

# Go To Public folder
echo "Changing into /public directory... "
cd public

# Add changes to git.
echo "Adding changes to git... "
git add .

# Commit changes.
echo "Committing changes to git... "
msg="Rebuilding site `date`"
if [ $# -eq 1 ]
then msg="$1"
fi
git commit -m "$msg"

# Push source and build repos.
echo "Pushing changes into /public directory... "
git push origin master

# Come Back up to the Project Root
echo "Change into root directory.."
cd ..

# Come Back up to the Project Root
echo -e "\033[0;32mDeploy Complete...\033[0m"

As a test, save some changes to your post (e.g. /blog/content/my-first-post.md)

Now run the deploy script

$ ./deploy.sh

HTTPS on GitHub Pages

GitHub Pages allow you host your site via HTTPS using the default domain or a custom domain. Here are a couple of quick reminders if you want to use a custom domain:

Update config.toml

To use a Custom Domain, be sure to update baseurl variable in your config.toml

baseurl = "https://<your_custom_domain>/"

Configure A Records

To use HTTPS and a Custom Domain you will Configure A Records with your DNS Provider.

Create a CNAME File

Create a CNAME file under static/CNAME (e.g. /blog/static/CNAME)

To use a custom domain for your GitHub Pages site, create a file static/CNAME. Your custom domain name should be the only contents inside CNAME. Since itâ€™s inside static, the published site will contain the CNAME file at the root of the published site, which is a requirements of GitHub Pages.

Wrapping Up

Ok, to quickly wrap things up. This post has introduced Hugo and walked through a few simple configurations for your Hugo project so that you can manage your source code in a private git repository and using git submodules build/deploy to a public GitHub Pages respository that hosts over HTTPS.

I hope that this gets you quickly up and running with your next static website.
